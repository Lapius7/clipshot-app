# clipshot-app

[English README is here](README.md)

Go言語で書かれたWindowsトレイ常駐クライアントです。ショートカットキーを押すと、その時クリップボードにある画像が自分の[clipshot-server](https://github.com/Lapius7/clipshot-server)インスタンスにアップロードされ、返ってきたURLがそのままクリップボードに書き戻されます — ShareXやGyazoと同じ「キーを押してリンクを貼り付ける」ワークフローを、自分で運用するサーバーに対して実現します。

- システムトレイに常駐し、通常はウィンドウが表示されません。
- アップロード方法は2種類: グローバルショートカットキー（クリップボード画像）と、トレイメニューの**Upload Image...**（ネイティブファイル選択、ディスク上の任意の画像）。
- ショートカットキーの組み合わせはSettingsダイアログから変更可能 — 設定ファイルを手動編集する必要はありません。
- APIトークンは平文でディスクに触れません — Windows資格情報マネージャー（DPAPIで保護）に保存されます。
- `https://` 以外のURLとは一切通信しません。
- 依存ツリーにCGOが一切無いため、Linux/WSLからクリーンにクロスコンパイルできます。

## これが存在する理由

これはClipShotプロジェクトのクライアント側です。サーバー側（[clipshot-server](https://github.com/Lapius7/clipshot-server)）がセルフホストのアップロードバックエンドで、このアプリは実際にWindowsマシン上に常駐し、ショートカットキーを監視してアップロードを実行する部分です。ホスト型のスクリーンショットツールを使うのではなくこのプロジェクトが存在する理由については、サーバー側のREADMEに全体像が書かれています。

## 動作の仕組み

独立した2つのアップロード経路があり、どちらも同じ「アップロード＋クリップボード書き込み」処理に合流します。

**ショートカットキー経路**（クリップボード画像）:
```
 1. 画像をクリップボードにコピー（CF_DIB形式、コピー or スクリーンショット）
 2. 設定したショートカットキーを押す（デフォルト: Ctrl+Shift+U）
 3. clipshot-appがクリップボードの画像を読み取り、PNGに再エンコード
 4. Windows資格情報マネージャーからAPIトークンをロード
 5. HTTPS経由で <instance-url>/api/upload にPNGをPOST
 6. サーバーが {"url": "https://.../i/<id>.png"} を返す
 7. clipshot-appがそのURLでクリップボードを上書き（プレーンテキストとして）
 8. 必要な場所にリンクを貼り付ける
```

**トレイメニュー経路**（ディスク上の任意のファイル）:
```
 1. トレイアイコンを右クリック → "Upload Image..."
 2. png/jpg/jpeg/gif/webpにフィルタされたネイティブファイル選択が開く
 3. clipshot-appが選択されたファイルを読み取り、コンテンツタイプを検出
 4. 上記のステップ4〜8と同一（トークンロード、アップロード、クリップボード書き込み）
```

アップロードを開始した後の全ステップは、一般的なブロードバンド環境であれば1秒未満で完了します。何か失敗した場合（クリップボードに画像が無い、ネットワークエラー、サーバーがアップロードを拒否など）は、トレイアイコンのツールチップに一時的に何が起きたかが表示されます（黙って何も起きないということはありません）。

## インストール・ビルド

現時点ではインストーラーはまだありません（[ロードマップ](#ロードマップ既知の制約)参照）— ソースからビルドするのが現在の方法です。

### 必要環境

- Go 1.23以上
- 結果を**実行する**ためのWindowsマシン。**ビルドする**ためにWindowsは不要です — このプロジェクトにCGO依存は一切無いため、Linux/macOS/WSLからのクロスコンパイルがそのまま動作します。

### ビルド

```bash
git clone https://github.com/Lapius7/clipshot-app.git
cd clipshot-app
CGO_ENABLED=0 GOOS=windows GOARCH=amd64 go build \
  -ldflags="-H windowsgui -s -w" \
  -o clipshot.exe ./cmd/clipshot/...
```

- `-H windowsgui` はコンソールウィンドウを抑制し、ターミナルがポップアップする代わりにきちんとしたトレイアプリとして振る舞わせます。
- `-s -w` はデバッグシンボルを取り除き、バイナリサイズを縮小します（見た目だけの調整、任意）。
- `CGO_ENABLED=0` がこのプロジェクトをWindows以外のホストからビルド可能にしている要因です — このプラットフォームでの依存（Windows版`getlantern/systray`、`golang.org/x/sys/windows`）はすべて純Goです。

`clipshot.exe` を対象のWindowsマシンの任意の場所にコピーして実行してください。

> **なぜインストーラー・コード署名がまだ無いのか?** 両方とも計画中です（Inno Setup＋実際のコード署名証明書でWindows SmartScreenに警告されないようにする）が、この初期骨格の範囲からは明示的に外しています。[ロードマップ](#ロードマップ既知の制約)参照。

## 初回セットアップ

1. `clipshot.exe` を実行します。トレイアイコンが表示されます — メインウィンドウはありません。
2. トレイアイコンを右クリック → **Settings**。
3. 順番に3つの値の入力を求められます:
   - **Instance URL** — あなたのclipshot-serverの公開アドレス。例: `https://img.example.com`。`https://` で始まる必要があり、それ以外は拒否されます。
   - **API Token** — サーバーの管理者が発行したトークン（`clipshot-server token create`、[clipshot-serverのREADME](https://github.com/Lapius7/clipshot-server/blob/main/README.ja.md#トークン管理)参照）。空欄のままにすると、そのインスタンスURLに既に保存されているトークンが維持されます。
   - **Hotkey** — クリップボードアップロードを実行するグローバルショートカットキーの組み合わせ。例: `Ctrl+Shift+U`（デフォルト）。`Ctrl`/`Alt`/`Shift`/`Win` のいずれか1つ以上＋1文字のキーで構成される必要があり、無効な組み合わせは保存前に拒否されます。
4. 保存すると、URLとショートカットキーはローカルの設定ファイルに書き込まれ、トークンはWindows資格情報マネージャーに書き込まれます。ショートカットキーは自動的に再登録されます — 再起動は不要です。

## 使い方

### ショートカットキー経由（クリップボード画像）

1. 画像をクリップボードにコピーします（スクリーンショット、ブラウザからコピーした画像など — `CF_DIB`データをクリップボードに置くものであれば何でも）。
2. 設定したショートカットキー（デフォルト `Ctrl+Shift+U`）を押します。
3. トレイのツールチップに結果が一時的に表示されます（成功時は `ClipShot: URL copied to clipboard`、失敗時はエラーメッセージ）。
4. 必要な場所に貼り付け（`Ctrl+V`）— すでにクリップボードにプレーンテキストとして入っています。

### トレイメニュー経由（任意の画像ファイル）

1. トレイアイコンを右クリック → **Upload Image...**。
2. ダイアログで `.png`/`.jpg`/`.jpeg`/`.gif`/`.webp` のいずれかのファイルを選択。
3. 上記と同じ結果になります — トレイツールチップが成功/失敗を表示し、URLがクリップボードに入ります。

### ショートカットキーの変更

トレイアイコンを右クリック → **Settings** で、3番目の入力欄に新しい組み合わせを入力します。新しいショートカットキーは即座に有効になります（古いものは先に登録解除されます）— 再起動は不要です。

トレイアイコンを右クリック → **Quit** でクリーンに終了します（ショートカットキーの登録も解除されます）。

## 設定・シークレットの保存先

| 内容 | 保存先 | 形式 |
|---|---|---|
| インスタンスURL、ショートカットキー組み合わせ | `%APPDATA%\ClipShot\config.json` | プレーンテキストJSON（このファイルにシークレットは含まれない） |
| APIトークン | Windows資格情報マネージャー、ターゲット名 `ClipShot:<instance-url>` | DPAPIで暗号化、現在のWindowsユーザーにスコープされる |

トークンは意図的に `config.json` から完全に排除されています — 誰かがそのファイルを読んだとしても（バックアップに入っていた、サポート問い合わせで送られた、スクリーンショットに写っていた等）、トークンはそこに含まれていません。インスタンスURLごとに個別の資格情報マネージャーエントリが作られるため、複数サーバーを設定しても以前のトークンが上書きされることはありません。

## セキュリティモデル

- **通信**: アップローダー（`internal/uploader`）は `https://` で始まらないインスタンスURLを強制的に拒否します — これを無効化する設定フラグはありません。サーバーがまだTLSの背後に無い場合は、まずそちらを修正してください（clipshot-serverのREADME参照）。
- **シークレットの保存**: APIトークンはWin32資格情報マネージャーAPI（`CredWriteW`/`CredReadW`）経由で保存され、現在のユーザープロファイル下でDPAPIによって暗号化されます。同じマシン上の他のWindowsユーザーアカウントからは読み取れず、このアプリ自身が書き込むどのファイルにも保存されません。
- **ネイティブUI、シェル呼び出し無し**: 設定ダイアログはPowerShellなどをシェルアウトしてプロンプトを表示するのではなく、生のWin32ウィンドウ/コントロールAPI（`CreateWindowExW`等）を直接使って構築されています — 入力経路のどこにも文字列補間によるコマンド実行は存在せず、一部の「お手軽な」Windowsダイアログライブラリが抱えるインジェクション系の脆弱性クラスを丸ごと回避しています。
- **CGO無し**: Windowsビルドの依存はすべて純Go（`golang.org/x/sys/windows` のsyscall、`getlantern/systray` の純Go Windowsバックエンド）です。これによりサプライチェーンとビルドプロセスがシンプルで監査しやすい状態に保たれます。
- **このアプリが守らないもの**: Windowsユーザーアカウント自体が侵害された場合、攻撃者は保存されたトークンを復号できます（DPAPIの復号はこのアプリ専用ではなく、ログインユーザーに紐づくため）。Windowsのログインは他の認証情報の境界と同様に扱ってください。

## トラブルシューティング

| 症状 | 想定される原因 |
|---|---|
| ショートカットキーを押しても何も起きない | 別のアプリケーションが同じグローバルショートカットキーの組み合わせを既に登録している。Settingsを開いて別の組み合わせ（例: `Ctrl+Alt+U`）に変更してください — 再起動は不要です。 |
| ツールチップに「clipboard has no image」と表示される | クリップボードに現在 `CF_DIB` データが入っていません。ほとんどのアプリ（ブラウザ、ペイント、スクリーンショットツール）からの画像コピーは動作しますが、（エクスプローラーなどからの）*ファイル*のコピーは通常対象外です。 |
| ツールチップに「no API token configured」と表示される | 設定がまだ保存されていない、または `config.json` に現在記載されているインスタンスURLとは異なるURLに対して保存されています。設定を再度開いてトークンを再入力してください。 |
| Settingsでショートカットキーが拒否される | 組み合わせは `Ctrl`/`Alt`/`Shift`/`Win` のいずれか1つ以上を `+` で結合し、最後に1文字のキーで終わる必要があります（例: `Ctrl+Shift+U`）。複数文字のキー（`Ctrl+Enter`、ファンクションキーなど）は現時点では未対応です。 |
| アップロードが401系のメッセージで失敗する | トークンが間違っている、失効済み、または現在指しているインスタンスURLと一致していません。サーバー側でトークンを再発行し、設定に再入力してください。 |
| アップロードがサイズ関連のメッセージで失敗する | 画像がサーバーの `MAX_UPLOAD_MB` を超えています。これはサーバー側の制限なので、運用者に上限を上げてもらうか、画像を縮小してください。 |

## プロジェクト構成

```
cmd/clipshot/main.go       エントリポイント: 設定ロード、ショートカットキー登録、トレイ起動
cmd/clipshot/icon.go       トレイアイコン（icon.ico）をバイナリに埋め込み
internal/config/           設定の読み書き（%APPDATA%\ClipShot\config.json）、URLバリデーション
internal/credstore/        Windows資格情報マネージャーのラッパー（CredWriteW/CredReadW/CredDeleteW）
internal/hotkey/           RegisterHotKeyによるグローバルショートカットキー登録、組み合わせのバリデーション
internal/clipboard/        クリップボードからCF_DIBを読み取りPNGに再エンコード、URLテキストを書き戻し
internal/uploader/         HTTPS限定のmultipartアップロードクライアント
internal/ui/               トレイメニュー（getlantern/systray）、純Win32ネイティブ設定ダイアログ、ネイティブファイル選択（GetOpenFileNameW）
internal/notify/           トレイツールチップの一時変更によるユーザーへのフィードバック
```

Windows以外向けのビルドは `_other.go` ファイル内のスタブ実装（「未対応」エラーを返すだけ）に対してコンパイルされます。これは単にどのOSからでもコードを検査・テストできるようにするためのもので、このアプリ自体はWindows上で動作することのみを意図しています。

## ダウンロードとリリースの検証

ビルド済みバイナリは[Releasesページ](https://github.com/Lapius7/clipshot-app/releases)で公開されています。**`clipshot.exe` を他の場所からダウンロードしないでください** — 公式ミラーも、サードパーティのダウンロードサイトも、自動アップデート機能もありません。このバイナリが他所でホストされているのを見つけた場合は、信頼しないでください。

### このバイナリはコード署名されていません

`clipshot.exe` は**コード署名証明書で署名されていません**（[ロードマップ](#ロードマップ既知の制約)参照）。これは次のことを意味します。

- 初回実行時にWindows SmartScreenが「認識されていないアプリ」という警告を表示する可能性が非常に高いです。これは小規模なOSSプロジェクトの未署名バイナリとしては想定通りであり、必ずしも改ざんの兆候ではありません — ただし、署名済みバイナリのようにWindows自体が発行者の身元を保証しているわけでもないということでもあります。
- 発行者の身元を署名チェーンで保証する代わりに、以下のチェックサム（および理想的には自分でソースからビルドすること）に依拠して、バイナリがこのリポジトリの生成物と一致することを確認する必要があります。

脅威モデル上、検証済みの発行者IDが必要な場合は、署名済みリリースが提供されるまでソースからビルドしてください（[インストール・ビルド](#インストールビルド)参照）。

### ダウンロードしたバイナリの検証

各リリースには `clipshot.exe` と一緒に `SHA256SUMS.txt` が含まれています。ダウンロード後、実行する前にハッシュが一致することを確認してください。

**PowerShell:**
```powershell
Get-FileHash .\clipshot.exe -Algorithm SHA256
```

**Linux/macOS/WSL:**
```bash
sha256sum clipshot.exe
# または、公開されているファイルと直接比較する場合:
sha256sum -c SHA256SUMS.txt
```

[Releasesページ](https://github.com/Lapius7/clipshot-app/releases)にある該当リリースの `SHA256SUMS.txt` の対応する行と結果を比較してください。完全に一致しない場合は、そのファイルを実行せず、再ダウンロードしてください。それでも不一致が続く場合はIssueを開いてください。

### リリースビルドを自分で再現する

ビルドは単純な`go build`（CGOなし、プラットフォーム特有のツールチェーンの癖もなし）なので、特定のタグのバイナリを再現してハッシュを自分で比較できます。

```bash
git clone https://github.com/Lapius7/clipshot-app.git
cd clipshot-app
git checkout v0.1.0   # ダウンロードしたリリースタグに合わせる
CGO_ENABLED=0 GOOS=windows GOARCH=amd64 go build -trimpath -ldflags="-H windowsgui -s -w" -o clipshot.exe ./cmd/clipshot/...
sha256sum clipshot.exe
```

Goのツールチェーンは現時点で全環境・全Goバージョンにわたるビット単位の再現可能ビルドを保証していないため、ここで不一致が出ても即座に危険信号というわけではありません — しかし一致した場合は、公開されているバイナリがこのソースツリーに対応しているという強力な独立した確認になります。

## ロードマップ・既知の制約

これは初期段階の骨格であり、完成したプロダクトではありません。既知のギャップを優先度順に挙げます。

- [ ] 単体の`.exe`ではなく、本格的なインストーラー（Inno Setup）
- [ ] コード署名（初回実行時にWindows SmartScreenが警告しないようにする）
- [ ] ログイン時の自動起動（レジストリRunキーまたはタスクスケジューラ登録）
- [ ] 複数文字のショートカットキー（ファンクションキー、Enterなど）への対応 — 現在は1文字のキーのみ
- [ ] ビルド検証・リリースアーティファクト用のGitHub Actions CI
- [ ] アップロード中の一時的なネットワーク障害に対するリトライ/バックオフ
- [ ] 自動テスト（現在の検証はクロスコンパイルチェックのみの手動確認 — コミット履歴参照）
